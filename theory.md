# Теоретичні основи системи автоматичної генерації військових документів

## 1. Великі мовні моделі (LLM)

Великі мовні моделі (Large Language Models, LLM) — це нейронні мережі на базі архітектури Transformer, навчені на мільярдах текстових токенів. Вони здатні розуміти природну мову та генерувати зв'язний текст у відповідь на запит (промпт).

У проєкті використовується модель **Qwen 2.5 (7B параметрів)** — квантована версія (4-bit, ~4.7 GB), розроблена компанією Alibaba Cloud. Модель виконується локально через **Ollama** без передачі даних до зовнішніх сервісів, що відповідає вимогам безпеки обробки службових документів.

### Основні параметри LLM-конфігурації

| Параметр | Значення | Пояснення |
|---|---|---|
| `temperature` | 0.2 | Низька температура → детермінований, формальний вивід |
| `numCtx` | 8192 | Розмір контекстного вікна (токени) |
| `timeout` | 180 с | Максимальний час очікування відповіді |

Низька температура (0.2) критично важлива для офіційного діловодства: модель обирає найімовірніші слова і не «фантазує», тоді як висока температура (> 1.0) давала б творчий, але непередбачуваний вивід.

## 2. Retrieval-Augmented Generation (RAG)

RAG — це архітектурний патерн, що покращує якість генерації LLM шляхом доповнення промпту релевантними фрагментами з бази знань (Knowledge Base). Це усуває дві основні проблеми «голої» LLM:

- **Галюцинації** — вигадані факти, яких не було у вхідних даних.
- **Дрейф формату** — відхилення від офіційного зразка документа.

### Процес RAG у системі

```
Вхідний запит (нотатки)
        │
        ▼
AllMiniLmL6V2EmbeddingModel
(перетворення тексту у 384-вимірний вектор)
        │
        ▼
Косинусна схожість з InMemoryEmbeddingStore
(> ~10 000 фрагментів із зразків і правил)
        │
        ▼
Відбір топ-2 фрагментів (score ≥ 0.45)
        │
        ▼
Передача у промпт LLM як [RAG]-контекст
```

### Джерела бази знань

| Файл | Зміст |
|---|---|
| `raport_rules.txt` | Правила складання рапортів за Наказом МО № 40 |
| `metodychna_rules.txt` | Вимоги до методичних розробок ВВНЗ |
| `metodychna_examples.txt` | Повні зразки: оборона взводу, медична підготовка, топографія |
| `metodychna_full_examples.txt` | Повні зразки: статутна, стройова, РХБ, фізична, інженерна, правова, зв'язок |
| `rag/generated_samples.txt` | Накопичені зразки згенерованих документів (Feedback Loop) |

### RAG Feedback Loop

Система реалізує механізм безперервного навчання: користувач може зберегти будь-який згенерований документ як новий зразок кнопкою «Зберегти як зразок». Цей фрагмент одразу індексується у векторній базі та впливає на наступні генерації. Це дозволяє системі адаптуватися до специфіки конкретного підрозділу з часом.

### Ліміти RAG-контексту

| Тип документа | Ліміт символів | Причина |
|---|---|---|
| Рапорт | 120 символів | Запобігає копіюванню імен і дат із чужих зразків |
| Методична розробка | 600 символів | Більший контекст для структурних зразків |

## 3. Мультиагентна архітектура

Система розподілена між спеціалізованими агентами за принципом **розподілу відповідальності** (Separation of Concerns). Кожен агент має власний системний промпт і виконує одну чітку задачу.

### Агент-аналітик (AnalystAgent)

**Вхід:** сирі нотатки → **Вихід:** JSON-структура

Промпт налаштований на витягування структурованих даних:
```json
{
  "documentType": "рапорт",
  "author": "сержант Кравченко Дмитро Юрійович",
  "recipient": "командир взводу лейтенант Поліщук О.В.",
  "subject": "відпустка 10 діб з 20 червня 2026"
}
```

Агент-аналітик свідомо відокремлений від агентів-редакторів: він не генерує текст, а лише розбирає вхід. Це зменшує плутанину між «розумінням задачі» та «виконанням задачі».

### Агент-редактор рапорту (RaportEditorAgent)

Налаштований на офіційний стиль військового діловодства зі строгими правилами:
- Перший рядок — завжди посада адресата
- Звернення «Прошу [що зробити], мені, [звання] [ПІБ]»
- Заборонені русизми: «отпуск» → «відпустка», «командировка» → «відрядження»
- Заборонені маркери: «невідомо», «[дужки]», мета-коментарі

### Агент-редактор методичної розробки (MethodychnaEditorAgent)

Налаштований на академічний стиль з обов'язковою структурою:
- Вступна / Основна / Заключна частина
- «Навчальна дисципліна» ≠ «Тема» (запобігає тавтологіям)
- Конкретний змістовий наповнення кожного пункту
- Використання «курсантів» замість «студентів»

## 4. Граматична перевірка (LanguageTool)

**LanguageTool** — відкрита бібліотека для перевірки граматики та орфографії з підтримкою більше 25 мов, включаючи Ukrainian (`uk`). Версія 6.4 містить актуальні правила для українського діловодства.

### Інтеграція в систему

```java
GrammarCheckService  →  JLanguageTool(new Ukrainian())
                     →  відключення STYLE + TYPOGRAPHY категорій
                     →  до 10 попереджень за перевірку
```

Категорії стилю та типографіки відключені навмисно: вони генерують суб'єктивні рекомендації, які не мають значення для офіційних документів. Залишені тільки граматичні та орфографічні помилки.

### Асинхронна ініціалізація

LanguageTool ініціалізується у фоновому потоці при старті сервера (`@PostConstruct`). Це запобігає затриці запуску (~3–5 секунд на завантаження моделі правил).

## 5. Формування документів (Apache POI + ДСТУ 4163:2020)

**ДСТУ 4163:2020** «Уніфікована система організаційно-розпорядчої документації» визначає вимоги до офіційних документів в Україні. **Наказ МО України № 40 від 31.01.2024** встановлює конкретні правила діловодства в Збройних Силах.

### Реалізація вимог стандарту

**Поля документа:**
- Ліве поле: 30 мм (для підшивки в справи)
- Праве поле: 15 мм
- Верхнє та нижнє: 20 мм

**Типографіка:**
- Шрифт: Times New Roman, 14pt
- Міжрядковий інтервал: 1.5
- Відступ абзацу: 1.25 см

**Структура рапорту:**
```
Адресат (посада, звання, ПІБ) — вирівнювання праворуч
[порожній рядок]
РАПОРТ — по центру, великими, жирний
[порожній рядок]
Текст звернення...
[порожній рядок]
підпис (таблиця 2 колонки): звання | ПРІЗВИЩЕ
```

**Структура методичної розробки:**
```
МЕТОДИЧНА РОЗРОБКА — по центру, великими, жирний
[порожній рядок]
Вид заняття: ...
Навчальна дисципліна: ...
...тіло документа...
```

## 6. Збереження даних (H2 + JPA)

Кожна генерація документа фіксується в H2 файловій базі даних (`data/docgen.mv.db`) через JPA-сутність `DocumentRecord`.

### Метадані генерації

| Поле | Тип | Призначення |
|---|---|---|
| `documentType` | String | Тип документа (рапорт, методична тощо) |
| `fileName` | String | Ім'я .docx файлу |
| `createdAt` | LocalDateTime | Час генерації |
| `processingTimeMs` | long | Час виконання конвеєру (мс) |
| `ragFragmentsUsed` | int | Кількість використаних RAG фрагментів |
| `ragContextFound` | boolean | Чи знайшов RAG релевантний контекст |
| `compareMode` | boolean | Чи це режим порівняння (з/без RAG) |

## 7. Веб-інтерфейс (SPA, Vanilla JS)

Фронтенд реалізований як одно-сторінковий застосунок на чистому JavaScript без фреймворків. Це мінімізує залежності та дозволяє розгортати систему без Node.js.

### Функціональність інтерфейсу

- Введення нотаток і вибір режиму (звичайний / порівняння)
- Відображення згенерованого документа в HTML-рендері
- Завантаження .docx
- Панель граматичних попереджень (LanguageTool)
- Перегляд RAG-контексту, що використовувався
- Завантаження файлів для парсингу (.docx / .pdf / .txt)
- Статистика: кількість генерацій, середній час, розподіл по типах

## 8. Особливості обробки

### Фільтрація RAG для методичних розробок

Векторна база містить фрагменти всіх типів документів. При генерації методичної розробки рядки, характерні для рапортів («доповідаю», «прошу надати», «відпустку» тощо), фільтруються ще до передачі в LLM. Це запобігає «забрудненню» методичних розробок рапортним стилем.

### Нормалізація типу документа

Модель іноді повертає тип англійською або з помилками: `"Raport"`, `"raport"`, `"rapport"` → нормалізується до `"рапорт"`. Це забезпечує правильне форматування DOCX незалежно від мови відповіді аналітика.

### Режим прямого форматування

Якщо вхідний текст вже є готовим документом (довжина > 500 символів і містить маркери готового документа), система пропускає LLM і безпосередньо форматує у .docx. Це економить час при завантаженні вже складених документів.

## 10. Безпека та конфіденційність

Система спроєктована для обробки службової (несекретної) документації:

- **Повністю локальний LLM** — модель `qwen2.5:7b` виконується на локальному комп'ютері, дані не залишають мережу підрозділу
- **Файлова база даних** — H2 зберігає метадані локально без зовнішніх з'єднань
- **Відсутність хмарних залежностей** — система працює без доступу до інтернету після початкового завантаження моделі
- **Векторна база в пам'яті** — `InMemoryEmbeddingStore` не персистується між запусками, знижуючи ризик витоку даних через базу знань
